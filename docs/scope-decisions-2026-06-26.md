## 1. スコープ絞り込み

- 当面は **代理店向け（operator）フローに集中**する。
- **traveler 向け UI・hotel staff 向け UI は保留**（今回のスコープ外）。
- 既存の**写真機能**（[lib/photos-store.ts](../lib/photos-store.ts)、`luggage-input` 系コンポーネント）は
  新運用では**使わない**。**撤去はせず一旦放置**する（コードは残すが運用導線から外す）。

> 補足: traveler 向けの導線は「配送トラッキング」（後述 §4）のみ復活する。
> ただしこれは認証なしのトラッキングトークン URL 方式であり、従来の traveler UI/会員登録とは別物。

---

## 2. 写真機能を廃止

- **旅行者・代理店ともに荷物写真は撮らない**。
- 運用は「**伝票を貼って終わり**」のシンプルなオペレーションとする。
- **月曜の DB 設計から、写真関連テーブル・ストレージ設計は除外**する。
- 写真を扱わないことで、**GDPR 等の個人情報リスクも大幅に軽減**される。

---

## 3. トラブル対応・補償の運用設計（未確定 — 月曜以降に詰める）

現時点では方針のみ。詳細は月曜以降に確定する。

- 当面は **「最強の対応」方針**（ケースバイケースで柔軟に対応する）。
- **窓口は代表者個人**（将来的に CS 体制化する前提）。
- 補償は **ヤマトの標準補償 + 運送保険の併用**を検討。
- DB 設計には **最小限の `claim_case` テーブルを含める**（将来拡張前提の最小構成）。

---

## 4. 配送トラッキング機能（新スコープ・必須）

Ship&co API から取得した配送状況を、**代理店と旅行者の両方**が見られるようにする。

### 代理店向け
- **operator 画面の拡張**として実装。
- **複数区間（leg）を横断して管理**できるビュー。

### 旅行者向け
- **認証不要**。**トラッキングトークン URL 方式**でアクセスする。
- バウチャー PDF に **URL / QR コードを記載**。
- **多言語対応・モバイル最適**。

### 取得方式
- **プル型（ポーリング）か ウェブフック型か は未確定** — 月曜の設計時に決定する。

### 関連テーブル
- `shipment_tracking`（Ship&co の配送状況キャッシュ）。
- トラッキングのアクセスログ 等。

---

## 5. 認証設計の方針

- **代理店ユーザー**: **Supabase Auth（email + password）**。
- **旅行者**: **認証なし**。長いランダムトークンの URL でアクセスする。
- 旅行者を BondEx に**会員登録させない**（インバウンド客の摩擦を回避するため）。
